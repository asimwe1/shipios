# ShipiOS Daisy Mobile Design

Use this skill when working on:

- the ShipiOS product UI if a mobile-native concept is needed
- any app concept generated by ShipiOS
- onboarding, dashboards, feeds, settings, checkout, profile, tabs, empty states, and flows that should feel like real iPhone screens

Do not use this skill for:

- desktop-first web pages
- generic marketing copy
- abstract design systems with no screen output

## Core rules

1. Daisy is for mobile screen design only.
2. Prefer Daisy runs over manual per-screen generation.
3. Reuse an existing Daisy project when continuing the same app or flow.
4. Save outputs in a way future agents can inspect: project id, run id, screen ids, rendered HTML, and screenshots when useful.
5. For ShipiOS-related work, keep the screens narrow, product-like, and believable. Avoid noisy “AI dashboard” layouts.

## ShipiOS-specific design direction

When using Daisy for ShipiOS or apps created by it:

- prioritize iPhone-native clarity over novelty
- keep one dominant action per screen
- use short copy and strong hierarchy
- design flows that can plausibly map into SwiftUI later
- avoid generic “card soup” and over-explained interfaces
- if the reference tone is Daisy / Rork / Appshots, bias toward prompt-first, proof-first, minimal navigation, and clean output surfaces

## Required environment

Read the API key from:

```bash
export DAISY_API_KEY="dsy_live_..."
```

Fail early if:

- `DAISY_API_KEY` is missing
- it starts with `dsy_test_`

Do not invent or stub a key.

## Default workflow

1. Reuse or create the Daisy project.
2. Start a run with a natural-language message describing the screen set.
3. Poll until the run reaches `succeeded`, `failed`, or `cancelled`.
4. Resolve `screenId`s from the run operations.
5. Fetch final screen HTML from Daisy.
6. Render screenshots when the user needs shareable proofs.
7. Save the project id, run id, and screen ids in the task notes or handoff.

## Reuse policy

If the user is continuing the same app:

- use the same Daisy project
- keep screen naming consistent
- describe edits relative to the existing screen labels

If the user is designing a new app:

- create a new Daisy project with a short memorable name
- record the new project id

## What to save for future agents

Whenever Daisy work is done, record:

- Daisy project name
- Daisy project id
- latest run id
- screen ids and labels
- where screenshots or exported HTML were saved

Update:

- `docs/status.md` if Daisy becomes part of the active workflow
- `.agents/handoff-template.md` output when handing work off
- `.codex/project-context.md` if a stable Daisy design practice changes
